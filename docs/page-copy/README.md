## 统一写法

- 主标题负责表达业务价值，不直接堆产品名词。
- 副标题负责补充业务含义、适用场景和结果方向。
- 段落正文优先解释品牌为什么值得做，而不是先解释系统怎么实现。
- 收益点只写品牌和运营团队能感知到的变化。

## 禁用倾向

- 过度技术化：如“接口”“数据回流”“自动化链路”“SCRM 沉淀”。
- 过度口语化：如“扫完就走”“先试试看”“后面没接住”。
- 过度空泛化：如“赋能增长”“全链路闭环”“数字化升级引擎”。
- 直接冒用他站真实客户、真实数据、真实服务承诺作为本站事实。

## 可借鉴但不可直接冒用

- 可借鉴：案例组织方式、结果表达方式、服务介绍结构、CTA 节奏。
- 不可直接冒用：他站客户名称、项目结果、公司地址、联系电话、邮箱、服务承诺。
- 正确做法：先按成熟官网结构写出本站终稿，再把本站真实客户、真实项目、真实数据替换进去。

## 内部占位使用规则

- 如需暂时借用他站客户名、数据和服务信息，只能在内部文案稿中以 `参考占位` 形式出现。
- 所有占位内容都必须带上 `上线前替换` 标记。
- 这类内容只能用于内部排版、设计和演示，不应直接作为对外官网事实发布。
- 内部占位信息统一存放在 [INTERNAL-placeholder-claims-do-not-ship.md](/Users/wangwang/Desktop/扫码官网/website/docs/internal/INTERNAL-placeholder-claims-do-not-ship.md)，不再混入正式页面文案稿目录。
